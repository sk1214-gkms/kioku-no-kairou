## 2. コアシステム（画面構成と操作）

画面遷移・スクロールを排除した1画面固定構成。

```
┌─────────────────────────────┐
│ [上部] ステージ番号 / メニュー(ヒント・タイトル) / タイマー※ │
├─────────────────────────────┤
│                             │
│ [中央] ドア + 壁・床                              │
│        interactables（タップ可能なホットスポット） │
│                             │
├─────────────────────────────┤
│ [下部] アイテムスロット / テキスト表示 / 入力パネル │
└─────────────────────────────┘
```

- **タイマー表示は timer モード時のみ**。
- 操作は **タップ（調査・選択）** と **入力パネル（数字 / 文字 / 順番タップ）** のみ。

## 3. ゲームモード仕様

| モード | 制限時間 | 失敗時の挙動 | 記憶への影響 | ヒント |
|---|---|---|---|---|
| ノーマル | なし | 解けるまで進行不可 | 全 full | あり（動画広告で解放） |
| ハード | なし | 解けるまで進行不可 | 全 full | **なし**。手がかりが架空言語『回廊文字』で表示され、『暗号解読書』で解読しないと解けない（→ 3.1） |
| タイマー | 1部屋 1〜2分 | 時間切れで強制的に次へ | 未クリア部屋は **corrupted（虫食い）** | あり |

> **重要**：モードは「記憶が欠けやすいか」に影響するだけ。結末を直接決めるのは
> 次章の `memory_score`（実際に何ステージ正解したか）である。
> （タイマーでも全問時間内に解けば memory_score=100% になり得る）

### 3.1 ハードモード：難化モディファイア・ツールキット（回廊文字ほか）

ハードは**「回廊文字だけ」ではなく、難化レバー（モディファイア）のツールキット**として設計する。
各部屋は `hard.modifiers` で使う難化を宣言し、エンジンはそれに応じて挙動を切替える（例：`cipher` のときだけ暗号解読書ボタンを表示）。
これにより**あらゆる仕掛けタイプにハード変種を用意でき**、1つの謎からノーマル/ハード両方を安く作れる。

| モディファイア | 効果 | 主な実現手段（現エンジン） |
|---|---|---|
| `cipher` | 手がかりを回廊文字化し解読書で変換 | `reveals_glyphs` / `label_glyph`（実装済） |
| `decoy` | 偽の手がかり/オブジェクトを追加（論理で除外） | `interactables` に `decoy:true` を追加 |
| `hidden_clue` | 平常見える手がかりを行動しないと見せない | `prerequisites` ＋ locked reveal（5号室方式） |
| `extra_step` | 入力前に前提操作を1つ増やす | `gimmick.prerequisites` |
| `scramble` | ボタン位置/ラベルを入替え、暗記・勘を無効化 | `interactables` の id 差し替え |
| `indirection` | 手がかり→さらに変換（反転/オフセット） | 解を変換後の値で定義（4号室方式） |
| `no_hint` | ヒント不可 | ハード標準 |

> 重要：**ハードのモディファイアは“難化レイヤー”**であって、仕掛けそのもの（drag / アイテム使用 等の基本ギミック）の実装とは別。
> 新しい謎を採用するには基本ギミックの実装が必要で、ハードはその上に上記レバーを重ねる。

#### cipher（回廊文字）の詳細

`cipher` モディファイアの手がかり表現は、**架空言語『回廊文字（Corridor Script）』**。定義は [`data/cipher.json`](../data/cipher.json)。

**仕組み**
- ハードでは、各ステージの手がかり（`reveals`）が回廊文字のグリフで表示される（例：`1024` ではなく `ᚠᚮᚢᚨ`）。
- プレイヤーはメニューから**『暗号解読書』**（cipher.json をブック UI で表示）を開き、グリフ → 数字/文字 に変換してから答えを入力する。
- これがノーマルに無い**追加の手数**であり、「解読書を見ながらでないと解けない謎」というコンセプトを満たす。

**ステージ別の手数設計**

| St. | ハードでの追加手数 |
|---|---|
| 1 | 絵画の数字をグリフ表示 → 解読して `1024` |
| 2 | 床の手がかり・壁ボタン**両方**をグリフ化 → 色を解読し青→黄→赤の順に押す |
| 3 | 時計文字盤をグリフ化 → 解読して `0345` |
| 4 | **二重**：鏡像のグリフを解読（8012）→ さらに反転を戻す（2108） |
| 5 | 暗闇に浮かぶ文字をグリフ化 → アルファベットに解読して `OPEN` |

**データ表現**（ハード上書きブロック内）
- 手がかり：`reveals_glyphs: ["g_1","g_0","g_2","g_4"]`（鏡像は `reveals_glyphs_reflected`）
- ボタン等のラベル：`label_glyph: "g_red"`
- 各ハードブロックに `decode_hint`（解読の方針メモ）を付与。

**運用方針**
- 解読書はハード時のみ常時閲覧可（ノーマル/タイマーでは非表示）。
- グリフは試作段階では `proto_symbol`（Unicode 文字）で代用し、製品では `glyph_image`（画像アセット）に差し替える。

## 4. 記憶充足度モデル（エンディングの土台）

### 4.1 記憶の状態
各ストーリーステージは、クリア結果に応じて1つの記憶（`memory_id`）の状態を確定する。

| 状態 | 条件 | 物語テキスト |
|---|---|---|
| `full` | 時間内に正解 | `text_full`（完全版） |
| `corrupted` | タイマーで時間切れ | `text_corrupt`（虫食い版） |
| `unknown` | 未到達 | 表示なし |

### 4.2 スコア
```
memory_score = (full の記憶数) / (全ストーリーステージ数) × 100
```
- 例：5ステージ中4つ full、1つ corrupted → memory_score = 80%
- このスコアと所持アイテムが **エンディング分岐の唯一の入力**（→ 第7章）。

### 4.3 隠しフラグ
| フラグ | 取得方法 |
|---|---|
| `has_culprit_evidence` | 特定ステージの **隠しホットスポット** をタップして遺留品を入手（第6章） |

## 5. データ構造（改訂版スキーマ）

v1 の単一 `correct_answer` を廃止し、**仕掛けタイプで分岐する判別ユニオン**＋**タップ領域（interactables）**を導入する。テキスト・ヒントは **ID参照** で外部化（多言語・校正対応）。

JSON Schema は [`data/stage.schema.json`](../data/stage.schema.json)、実データは [`data/stages/`](../data/stages/) 参照。

### 5.1 ステージ定義（共通構造）
```jsonc
{
  "stage_id": 1,
  "assets": { "background": "...", "door": "...", "bgm": "..." },

  // 画面上のタップ可能オブジェクト（=画面レイアウトにそのまま使える）
  "interactables": [
    { "id": "painting", "rect": [x, y, w, h], "zoom_image": "...", "reveals": "1024" }
  ],

  // 仕掛け本体（type で必要フィールドが変わる）
  "gimmick": {
    "type": "number_pad | text_input | sequence_tap | state_toggle",
    "prerequisites": [ { "interactable": "breaker", "state": "off" } ], // 任意：前提状態
    "solutions": { "normal": "...", "hard": "..." },
    "validate": "exact | case_insensitive | order"
  },

  // ハードで盤面ごと変える場合の上書き（任意）
  "hard": { "interactables": [...], "gimmick": {...}, "hints": [...] },

  "hints": ["text_id_1", "text_id_2"],        // ID参照
  "rewards": {
    "memory_id": "mem_01",
    "text_full":    "t_s01_full",             // ID参照
    "text_corrupt": "t_s01_corrupt",          // ID参照
    "item": null                              // 取得アイテムID or null
  }
}
```

### 5.2 仕掛けタイプ（判別ユニオン）

| type | 説明 | 主なフィールド | 例 |
|---|---|---|---|
| `number_pad` | 数字ダイヤル/テンキー | `solutions`(文字列) | St.1, St.3 |
| `text_input` | 文字入力 | `solutions`(文字列), `validate:case_insensitive` | St.5 |
| `sequence_tap` | 順番にタップ | `solutions`(id配列), `validate:order` | St.2 |
| `state_toggle` | オブジェクト状態を変える | `prerequisites` + 後続入力 | St.5(ブレーカー) |
| `condition` | 前提が揃うと自動クリア（入力なし） | `prerequisites` | 鍵/ドアノブで開ける系 |
| `drag` | ピースを枠へドラッグして組む | `solutions`(piece→slotのMap) | St.7(図面) |

**アイテム使用（item_use）**：これは gimmick タイプではなく interactable の能力。`requires_item` で「使う」のに必要なアイテムを指定し、`on_use_state`（前提を満たす）/`on_use_gives_item`/`on_use_reveals` で効果を定義する。St.6（ドライバーでパネルを外す）で使用。

> **複合仕掛け**は `prerequisites`（前提となる interactable の状態）＋主 `type` で表現する。
> 例：St.5 = ブレーカーを off にする（prerequisite）→ 文字入力 "OPEN"。

### 6.1 隠しアイテム入手経路（Ending C の前提）— 新規設計
- **配置**：ステージ4「鏡の反転」。謎解きには不要な隠しホットスポット `hidden_drawer`（鏡台の引き出し）を追加。
- **入手物**：`item_culprit_evidence`（真犯人の遺留品）。
- **設計意図**：解法に無関係なため**じっくり探索した人だけが発見**する。タイマーモードでは探索の余裕がなく取り逃しやすい → 「丁寧に遊んだ人だけが Ending C に到達できる」公平な隠し要素になる。

## 7. 最終ステージ（第30号室）とエンディング分岐 — ロジック確定

### 7.1 方式：選択 × 状態のマトリクス（C案）
- 3つの選択肢は **常に表示**（封印しない）。
- 選んだ選択肢が**結末の系統**を決め、`memory_score` と所持アイテムが**その系統内のどの結末か**を決める。
- これにより「自由な選択」と「丁寧に遊んだ人が報われる公平さ」を両立する。

画面テキスト：「外からサイレンが響く。私は一体、何者なんだ？」

### 7.2 分岐マトリクス

| 選択肢 | 条件 | エンディング | 種別 |
|---|---|---|---|
| ① 警察に通報（受話器） | `deduction_correct` かつ `memory_score ≥ 70` | **A＋：真・無実の証明**（推理成功・証拠提出） | トゥルー（完全） |
| | `memory_score ≥ 70` | **A：無実の証明** | トゥルー |
| | `memory_score < 70` | **A′：疑惑の保護**（証拠不足で容疑者扱い） | ビター |
| ② 暖炉へ逃げる（暖炉） | `memory_score < 40` | **B：終わらない逃亡**（St.1へループ） | バッド |
| | `memory_score ≥ 40` | **B′：逃亡の代償**（逃げる必要はなかった） | ビター |
| ③ 裏口から出る（裏口） | `has_culprit_evidence = true` | **C：共犯者の告発** | 隠しビター |
| | `has_culprit_evidence = false` | **C′：ただの逃走**（鍵は開くが何も解決しない） | ノーマル |

- 閾値（70 / 40）は [`data/endings.json`](../data/endings.json) で調整可能なデータとして外出し。
- **設計効果**：
  - ノーマル/ハードで全記憶 full → ①で**A（トゥルー）**に到達。
  - タイマーで記憶が虫食いだらけ → 主観的に「自分が犯人」と思い込み②へ → **B（ループ）**。これがコンセプト通りの“焦燥が結末を変える”体験。
  - 隠し遺留品を持つ探索派のみ③で**C**。

### 7.3 最終問題（推理ステップ）— 採用：任意・加点式

**方針**：必須ゲートではなく、**任意・加点式の推理ステップ**として実装する。データは [`final_room.json`](../data/stages/final_room.json) の `deduction` ブロック。

**仕掛け**
- 最終室で3択の前に「記憶を整理する（任意）」を表示し、1問だけ問う：
  「集めた記憶から、自分は何者だと結論づける？」
- 正解＝「館の主に招かれ、別の何者かの凶行に巻き込まれた**目撃者**」。これは **St.2（招待状）と St.4（日記）が `full`** で初めて論理的に成立する。
- 該当記憶が `corrupted` の場合、その選択肢の根拠は虫食い（`■■`）で示され、**論理的に選べない**＝「記憶が壊れているから自分が何者か分からない」という物語とゲーム性が一致する。
- 正解すると `deduction_correct = true`。① 通報の結末が **A＋（真・無実の証明）** に格上げ。
- スキップ可。答えない/間違えても `memory_score` で通常どおり分岐する（詰み・理不尽なし）。

**設計判断の根拠**
- 必須ゲートにすると、タイマーで虫食いの人が真エンドから救済なく締め出され、`memory_score` モデルと二重管理になるため不採用。
- 任意・加点式なら、推理が本物の仕掛けになりつつ、コアループから独立して**後付け可能**（→ プロトは MVP を先に出し、本機能は第2段で追加）。

## 9. 横断仕様 / 運用

- **テキスト外部化**：全文言を [`data/text_ja.json`](../data/text_ja.json) に ID で集約。将来の多言語は `text_en.json` 追加で対応。
- **広告頻度**：全ステージ間ではなく **Nステージごと**＋初回プレイは抑制（フリークエンシーキャップ）。値は設定で調整。
- **アクセシビリティ**：St.5「暗闇入力」は最低限のコントラスト確保。タイマーに一時停止（ポーズ）を用意。
- **規模感**：本書はプロト（5ステージ＋最終室）まで。製品版「第30号室」へ拡張時も同スキーマで追加可能。

## 11. Flutter 実装メモ

技術スタックは **Flutter** に決定。本データ群はエンジン非依存なので、`assets/data/` に配置して読み込む。

**推奨パッケージ**
- 状態管理：`riverpod`（または `provider`）。セーブ状態（[`save_template.json`](../data/save_template.json) 相当）を1つの State で持つ。
- 永続化：`shared_preferences`（軽量）または `hive`（構造化）。
- 広告：`google_mobile_ads`（リワード動画＝ヒント解放、インタースティシャル＝ステージ間）。
- 効果音/BGM：`audioplayers` か `just_audio`。

**画面/ウィジェット構成（1画面固定）**
- `GameScreen`（Stack 構成）
  - 背景（`assets[background]`）＋ ドア（`assets[door]`）
  - `interactables` を `Positioned`(rect) + `GestureDetector` でタップ領域化
  - 上部バー（ステージ番号 / メニュー / タイマー※）、下部（アイテムスロット / テキスト / 入力パネル）
- 入力パネルは `gimmick.type` で出し分け：`number_pad` / `text_input` / `sequence_tap`（タップ順記録）/ `state_toggle`（前提状態の管理）。
- `CipherBookOverlay`：ハード時のみメニューから開く解読書（[`cipher.json`](../data/cipher.json) を表示）。
- `FinalRoomScreen`：`deduction`（任意推理）→ 3択 → [`endings.json`](../data/endings.json) のルール評価で結末表示。

**実装順（スコープ管理）**
1. MVP：データ読み込み → St.1〜5（ノーマル）→ `memory_score` による A/B/C 分岐。
2. モード追加：ハード（回廊文字＋解読書）、タイマー（虫食い）。
3. 加点機能：最終問題（7.3）、隠しアイテム（Ending C）、広告。
